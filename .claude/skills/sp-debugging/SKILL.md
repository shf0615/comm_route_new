# 系统化调试

## 核心原则

**先找根因，再修复。修复必须通过 TDD：写失败测试复现 bug → 最小修复 → 绿灯。**

**启动声明：** "我正在使用 debugging 技能来系统化定位问题。"

## 铁律

```
没有根因调查，不允许提出修复。
没有失败测试复现，不允许改代码。
```

## 四阶段流程

### Phase 1: 根因调查

1. **完整阅读错误信息** — 栈追踪、行号、错误码
2. **稳定复现** — 能可靠触发？步骤？不可复现则收集更多数据
3. **检查最近改动** — git diff、最近 commit
4. **追踪数据流** — 坏值从哪来？追溯到源头

### Phase 2: 模式分析

1. 找到工作的例子（代码库中类似能工作的代码）
2. 对比差异
3. 理解依赖

### Phase 3: 假设与验证

1. 形成单一假设，写下来
2. 最小改动验证
3. 一次一个变量
4. 有效 → Phase 4；无效 → 新假设

### Phase 4: TDD 修复

1. **Red:** 用失败测试复现 bug

**若已有测试精确复现了 bug（运行失败且失败原因就是该 bug），直接用它作为 Red，无需重复编写。**

若无现成测试：

```python
def test_bug_reproduction():
    # Given [触发 bug 的条件]
    # When [触发动作]
    # Then [期望的正确行为，当前会失败]
```

2. **Green:** 最小修复让测试通过
3. **Refactor:** 清理（如需要）
4. **验证:** 运行全部测试确认无回归

### 3 次修复失败规则

3 次修复都没解决 → 停止，这可能是架构问题，与用户讨论。

## 危险信号

| 想法 | 正确做法 |
|------|----------|
| "先快速修一下" | 运行调查流程 |
| "改一下看看行不行" | 形成假设再验证 |
| "一次改多个地方" | 一次一个变量 |
| "我大概知道在哪" | 看到症状≠理解根因 |
