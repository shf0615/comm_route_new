# 头脑风暴与行为规格设计

将想法转化为经过验证的行为规格和技术方案，通过自然协作对话完成。

先理解当前项目上下文，然后逐个提问细化想法。理解要构建什么后，逐段呈现设计并获得用户认可。

<HARD-GATE>
在行为规格和技术方案经过审查并获得用户明确确认之前，不得写入任何实现代码、创建项目骨架、或调用任何实现技能。无论项目多简单，都必须走完流程。
</HARD-GATE>

## 反模式："这太简单了不需要设计"

每个项目都走这个流程。一个 todo list、一个单函数工具、一个配置变更——全部如此。"简单"项目正是未经审视的假设造成最多浪费的地方。设计可以很短（真正简单的项目几句话），但你必须呈现它并获得认可。

## Checklist

必须按顺序完成：

1. **探索项目上下文** — 查看文件、文档、最近 commit
2. **逐个提问** — 一次一个，理解目的/约束/成功标准
3. **提出 2-3 种方案** — 附带权衡和推荐
4. **逐段呈现设计** — 按复杂度调整篇幅，每段确认后再继续
5. **写入规格文档** — .feature + -design.md，提交 git
6. **规格自审** — 快速检查占位符、矛盾、歧义、范围（见下方）
7. **对抗性审查** — 派遣子agent独立审查，循环直到无严重问题
8. **用户审阅** — 请用户确认规格文件，有修改则改后重审
9. **转入实现** — 调用 `sp-writing-plans`

## 流程

### 理解想法

- 先查看当前项目状态（文件、文档、最近 commit）
- 在问详细问题之前，先评估范围：如果请求描述了多个独立子系统（如"构建一个有聊天、文件存储、计费和分析的平台"），立即标记。不要花时间细化一个需要先分解的项目。
- 如果项目太大无法用一份 spec 覆盖，帮用户分解为子项目：各部分是什么、它们如何关联、应以什么顺序构建？然后对第一个子项目走完整设计流程。每个子项目有自己的 spec → plan → implementation 循环。
- 对范围合适的项目，逐个提问细化想法
- 优先使用选择题，开放式也可以
- 每条消息只问一个问题 — 需要更多探索就分多条
- 聚焦于理解：目的、约束、成功标准

### 探索方案

- 提出 2-3 种不同方案，附带权衡分析
- 以对话方式呈现选项，带上你的推荐和理由
- 以推荐方案开头，解释为什么推荐

### 呈现设计

- 确信你理解要构建什么后，呈现设计
- 按复杂度调整每段篇幅：简单的几句话，复杂的最多 200-300 词
- 每段呈现后问是否正确
- 覆盖：**行为规格（BDD Scenarios）**、架构、组件、数据流、错误处理、测试
- 准备好回头澄清（如果某部分说不通）

**段落顺序：**
1. **行为规格** — 先呈现核心 Scenarios（正常路径+异常路径），确认行为预期正确
   - ≤2 个 Feature：一次性呈现全部 Scenarios
   - \>2 个 Feature：按 Feature 分次呈现，每个 Feature 单独确认
2. 架构与组件
3. 数据流与关键接口
4. 错误处理
5. 测试策略

**设计隔离性：**
- 将系统拆为更小的单元，每个有一个清晰目的，通过明确接口通信，可独立理解和测试
- 对每个单元能回答：它做什么？如何使用它？它依赖什么？
- 能否不看内部实现就理解一个单元？能否修改内部实现不破坏消费者？如果不能，边界需要调整。
- 更小、边界清晰的单元也更容易处理——能在上下文中一次看完的代码，推理更可靠，编辑更精确。文件变大时，往往说明它做了太多事。

**在现有代码库中工作时：**
- 探索现有结构后再提议修改。遵循现有模式。
- 如果现有代码有影响当前工作的问题（文件过大、边界不清、职责纠缠），作为设计的一部分包含针对性改进——就像优秀开发者改进他们正在工作的代码一样。
- 不提出无关重构。聚焦于服务当前目标的改进。

## 产出格式

### 行为规格（`.feature` 文件）

```gherkin
Feature: [功能名称]
  [一句话描述功能目的]

  Background:
    Given [所有场景共享的前置条件]

  Scenario: [正常路径 - 场景名]
    Given [前置条件/上下文]
    And [额外条件]
    When [触发动作]
    Then [预期结果]
    And [额外断言]

  Scenario: [异常路径 - 场景名]
    Given [前置条件/上下文]
    When [触发错误动作]
    Then [预期错误处理]

  Scenario Outline: [参数化场景名]
    Given [前置条件]
    When [动作 with <param>]
    Then [预期结果 <expected>]

    Examples:
      | param   | expected        |
      | value1  | result1         |
      | value2  | result2         |
      | 边界值  | 边界结果        |
```

**规格要求：**
- 每个 Feature 至少 1 个正常路径 + 1 个异常路径 Scenario
- 有多组输入/输出时使用 Scenario Outline + Examples
- Scenario 用业务语言描述，不涉及实现细节
- 每个 Then 必须可验证（可转化为 assert）

### 技术方案（`-design.md` 文件）

按复杂度调整每部分篇幅：简单的几句话，复杂的详细展开（每部分最多 200-300 词）。

覆盖：架构、组件、数据流、错误处理、测试。

## 设计完成后

**文档：**
- 行为规格写入 `docs/specs/YYYY-MM-DD-<topic>.feature`
- 技术方案写入 `docs/specs/YYYY-MM-DD-<topic>-design.md`
  - （用户对规格位置有偏好时，遵循用户偏好）
- 提交 git

**规格自审：**
写完文档后，以新鲜眼光审视：

1. **占位符扫描：** 任何 TBD/TODO/待定/不完整的部分？修正。
2. **内部一致性：** 技术方案与行为规格是否矛盾？
3. **范围检查：** 是否聚焦到可以一个实现计划覆盖？还是需要分解？
4. **歧义检查：** 任何 Given/When/Then 是否有多种解读？明确选一种。
5. **可测试性：** 每个 Then 是否可直接转化为 assert？

发现问题直接修正，无需重审——直接改然后继续。

**对抗性审查：**
派遣独立子agent，使用 `sp-adversarial-review` 作为角色指令。提供：行为规格 + 技术方案。

循环规则：
- 有严重问题 → 修正后重新提交审查
- 无严重问题 → 终止迭代
- 最多 3 轮

**用户审阅：**
> "规格和技术方案已写入 `<path>`。请审阅，有修改意见告诉我，确认后进入实现计划。"

等待用户回复。有修改则改后重跑审查。确认后进入下一步。

**实现：**
- 调用 `sp-writing-plans` 创建 TDD 实现计划
- 不调用任何其他技能。sp-writing-plans 是下一步。

## 关键原则

- **一次一个问题** — 不要一次问多个
- **选择题优先** — 比开放式更容易回答
- **YAGNI** — 移除所有不必要的功能和场景
- **探索替代方案** — 始终提出 2-3 种方案再决定
- **逐段确认** — 分段呈现，每段获得认可再继续
- **灵活回退** — 随时可以回头澄清
