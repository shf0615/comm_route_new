# 并行子Agent派遣

## 概述

将独立任务分配给子agent并行执行。

## 何时使用

- 2+ 个独立任务，无共享状态，不会互相干扰

## 何时不使用

- 任务相关联
- agent 会编辑同一文件
- 需要全系统上下文

## 流程

1. **识别独立域** — 按问题域分组，确认各域独立
2. **构造 Prompt** — 使用下方模板
3. **并行派遣** — 同一消息中派遣多个agent
4. **审查整合** — 全部返回后：验证无冲突，运行全部测试
5. **失败处理** — 见下方

## Prompt 模板

```
## 任务
[一句话描述目标]

## 范围
- 只修改: [文件列表]
- 不要动: [排除的文件/模块]

## 上下文
[必要的背景信息、相关代码片段、行为规格摘要]

## 约束
- [行为变更] 遵循 Red-Green-Refactor：先写失败测试，再实现
- [非行为变更（重构/注解/格式）] 修改后运行全部测试确认无回归即可

## 输出要求
返回：
1. 修改摘要（改了什么、为什么）
2. 测试运行结果
3. 遗留问题（如有）
```

## Prompt 要求

- **聚焦** — 一个明确问题域
- **自包含** — 所有需要的上下文都提供
- **输出明确** — agent 该返回什么
- **TDD 约束按性质区分：**
  - 涉及行为变更（新功能、bug 修复）→ 要求 Red-Green-Refactor
  - 非行为变更（重构、类型注解、格式化、文档）→ 只需修改后全部测试通过

## 失败处理

子agent返回失败时：

| 失败类型 | 处理 |
|----------|------|
| 测试不通过 | 检查是 agent 代码 bug 还是任务冲突。单任务 bug → 重新派遣修复。任务冲突 → 合并为一个任务重做 |
| agent 被阻塞 | 补充上下文后重新派遣 |
| agent 修改了不该动的文件 | 回滚该 agent 的修改，重新派遣并强调约束 |

## 验证

agent返回后运行全部 Scenario 测试，确认所有修改协同工作。
