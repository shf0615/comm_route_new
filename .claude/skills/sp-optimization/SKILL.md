# 代码优化

## 概述

在测试绿灯保护下优化代码。优化不能破坏任何行为。

**启动声明：** "我正在使用 sp-optimization 技能来优化代码。"

## 前置条件

- 行为规格文件（`.feature`）— 作为行为守护基线
- 技术方案（`-design.md`）— 作为架构约束锚点
- 已通过全部测试的代码

## 流程

```
用户选择优化范围
  → 派遣优化子agent
  → 子agent执行优化
  → 运行全部测试（Scenario 测试必须全过）
  → 有测试失败？→ 回滚该项优化
  → 有恶性问题残留？→ 继续迭代
  → 无恶性问题 + 测试全过 → 结束
```

## Step 1: 用户配置

可多选：
1. **架构** — 模块划分、耦合度、重复代码
2. **Bug/安全** — 潜在bug、资源泄漏、安全漏洞
3. **性能** — 运行效率、内存、响应速度
4. **可维护性** — 可读性、命名、代码组织

## Step 2: 优化执行

子agent职责：
- 识别问题（恶性/非恶性分类）
- 执行优化
- 每项优化后运行测试确认绿灯

**恶性问题：** 功能bug、架构腐化、安全漏洞、资源泄漏

## Step 3: 测试守护

**铁律：优化后测试必须全部通过。**

- 测试失败 → 回滚该优化项
- 如果优化需要改变行为 → 停止，与用户确认是否更新行为规格

## Step 4: 循环

还有恶性问题 → 继续迭代（聚焦恶性问题，不发散）
无恶性问题 + 绿灯 → 结束

## 约束

- 优化不能破坏任何 Scenario 测试
- 优化不能偏离技术方案的架构意图
- 行为变更必须先更新行为规格和测试
